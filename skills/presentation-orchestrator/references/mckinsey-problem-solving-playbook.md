# McKinsey Problem-Solving Playbook

Use this reference when the user asks for McKinsey-style consulting thinking, problem-solving decks, ambiguous business questions, scenario options, issue trees, or references such as `The McKinsey Mind`, `麥肯錫的專業思維`, `麥肯錫問題分析與解決技巧`, or the user's shorthand `這才是麥肯錫解決問題的方法`.

## Problem-Solving Spine

Use this spine before the slide plan when the situation is messy.

1. Classify the problem.
   - Restore current state: performance dropped, defect emerged, target missed.
   - Prevent potential issue: risk may materialize, market may shift, execution may fail.
   - Pursue ideal state: growth, transformation, new market, future operating model.
2. Convert the problem into an issue question.
   - Bad: "Growth is slow."
   - Better: "Should we invest in vertical GTM to restore efficient growth over the next two quarters?"
3. Build a MECE issue tree.
   - Split by drivers that can change the answer: market, customer, offer, channel, economics, capability, risk.
   - Prioritize branches by impact and uncertainty.
4. Define alternatives and scenarios.
   - Compare at least two real options plus "do nothing / defer" when leaders must choose.
   - Use scenario logic when outcomes depend on uncertain adoption, competition, cost, or timing.
5. Create the analysis workplan.
   - State what evidence would prove, weaken, or reverse the recommendation.
   - Separate source-backed facts, user-provided facts, assumptions, and inference.
6. Synthesize the "so what".
   - Convert findings into decision implications, not only observations.
   - Name what the executive should approve, reject, pause, test, or fund.
7. Prepare stakeholder handling.
   - Anticipate sponsor concern, skeptical executive objection, operating owner feasibility concern, and data-review challenge.

## Slide Patterns

| Situation | Recommended exhibit | Action-title pattern |
|---|---|---|
| Ambiguous problem | Problem classification + issue question | "The decision is not whether growth slowed, but which recovery path we fund" |
| Need decomposition | MECE issue tree | "Three branches explain whether vertical GTM can restore efficient growth" |
| Need evidence plan | Analysis workplan | "The next two evidence checks determine whether we invest or pause" |
| Need choice | Option comparison / decision table | "Vertical GTM wins only if pipeline quality offsets setup cost" |
| Need scenario handling | Scenario matrix | "The recommendation holds in base case but reverses if payback exceeds threshold" |
| Need final synthesis | So-what / now-what card | "Approve a gated vertical pilot, not a full-scale GTM rebuild" |

## Consulting Questions

- What type of problem is this: restore, prevent, or pursue?
- What is the exact issue question leaders need answered?
- Which issue-tree branches are mutually exclusive and collectively useful enough for this decision?
- Which two or three options are genuinely available?
- What evidence would change our mind?
- What is the "so what" for budget, priority, sequencing, ownership, or risk?
- What objection will the smartest skeptic raise first?

## Traps

- Using "McKinsey-style" as a brand signal instead of a problem-solving discipline.
- Building a complete issue tree without prioritizing the branches that change the decision.
- Presenting options without a default recommendation.
- Calling a claim "fact-based" when the evidence ledger marks it as assumption or inference.
- Hiding the "do nothing" alternative when it is a real executive option.
- Jumping from framework diagnosis to slides before forming the consulting POV.
