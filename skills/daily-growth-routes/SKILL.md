# Daily Growth Routes

Use this skill when the task is to run or design a daily 6DuckLearn growth review across community traffic and product marketing. It keeps AI/RSS community scouting separate from skills-led workflow demos.

## Operating Rules

- Keep every output private, draft-only, and approval-required.
- Do not publish, schedule, send externally, write to community database tables such as `shared_items`, or call external posting APIs.
- Do not claim adoption, virality, revenue, customer proof, rankings, awards, or guaranteed growth unless checked evidence proves it.
- If evidence is thin, label the output as a hypothesis, demo, or internal workflow.
- Treat the workflow as a growth learning loop, not a generic content generator.

## Route A: AI/RSS Community Scout

1. Select exactly 5 fresh AI/RSS articles when available.
2. Score each usable article from 1-5 on:
- builder relevance
- freshness
- novelty or tension
- 6DuckLearn angle
- community conversation potential
3. Produce 5 source cards with feed/source, title, URL, thesis, audience, score, evidence, and why it matters to 6DuckLearn members.
4. Draft 1-3 approval-required community posts from the highest scoring articles or clusters.
5. For each draft, include a title, member-style comment, summary, suggested channel/community, tags, source URLs, CTA, first-reply idea, do-not-post-if rule, claim ledger, and approval status.

Read [route-a-community-scout-template.md](./references/route-a-community-scout-template.md) when producing Route A output.

## Route B: Skill + Codex Workflow Demo Builder

1. Select one critical workflow problem and one relevant 6DuckLearn skill/Codex stack.
2. By default, call the output a Skill + Codex workflow demo or use-case draft.
3. Use "success case" only when there is evidence from real usage, shipped product behavior, or a documented internal run.
4. Include target user, before-state pain, workflow, steps, expected artifact, proof level, limitations, and next action.
5. Mark promo copy for blog, community, or skill-page reuse as approval-required.

Read [route-b-workflow-demo-template.md](./references/route-b-workflow-demo-template.md) when producing Route B output.

## Skill Availability

- Verify each named skill before presenting it as packaged, public, or live.
- If `content-summarizer` or another named skill is unavailable, mark it as missing or local fallback.
- Track website gaps separately: `go-to-market`, `pkm-synthesis`, and `domain-product-research` may exist as packaged skills without being live indexed website pages.

## Required Output

1. Run metadata: date/time, article count, route status, failed feeds.
2. Route A source cards with scoring values.
3. Route A community drafts with distribution plan and approval status.
4. Route B Skill + Codex workflow demo with proof level and limitations.
5. Skill stack used and availability status: public pack, website skill page, local fallback, or missing.
6. Claim ledger with evidence, confidence, and approved-safe wording.
7. Smoke-test evidence: live route health, article count, feed failures, website skill-page gaps.
8. 7-day learning metrics.
9. Next actions for community sharing, product marketing, SEO/GEO, or website skill sync.

Read [claim-ledger-and-metrics-template.md](./references/claim-ledger-and-metrics-template.md) for the claim ledger and metric fields.

## Language

Use:
- Skill + Codex workflow demo
- curated model-agnostic skill templates
- Codex-ready repeatable workflow
- draft GTM workflow
- approval-required community draft

Avoid unless evidence-backed:
- viral
- proven growth
- growth engine
- top by adoption
- customer success
- traffic guaranteed
